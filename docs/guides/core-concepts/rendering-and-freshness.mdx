## Overview

Waku gives you three separate choices: prerender static UI at build time, render dynamic UI for each request, and optionally cache selected work. Dynamic rendering is not implicitly cached by Waku.

This page is the reference for how those choices interact and for the cache layers that exist around them. For a hands-on introduction, see [Static and Dynamic Rendering](/guides/static-and-dynamic-rendering) in the Learn series.

## The layers

| Layer                      | When server code runs            | Freshness                  | Invalidation                            |
| -------------------------- | -------------------------------- | -------------------------- | --------------------------------------- |
| Static rendering           | At build time                    | Fixed until the next build | None; a rebuild produces a new artifact |
| Dynamic rendering          | On every request                 | Fresh by default           | None needed; nothing is cached          |
| Explicit application cache | On a cache miss                  | Defined by your policy     | Explicit, through your cache            |
| Client navigation reuse    | In the browser during navigation | A navigation optimization  | Separate from server data               |

These layers are independent. The first two are Waku's rendering modes. The third is an optimization you may add. The fourth is how the client router makes navigation fast, and it never affects what the server computes.

## Static rendering

Static is the default for pages, layouts, and slices. The output is produced once, during the production build, and served unchanged until the next build replaces it. A rebuild is not "cache invalidation"; it is producing a new artifact.

```tsx
// ./src/pages/built.tsx
export default async function BuiltPage() {
  const builtAt = new Date().toISOString();

  return <p>This page was prerendered at {builtAt}.</p>;
}
```

The timestamp above is fixed at the moment the build ran and identical for every visitor.

Static fits content that is the same for every visitor and changes only when you deploy: marketing pages, documentation, catalogs, blog posts. Because static rendering happens at build time, request-specific state does not exist for it; do not read headers or [request context](/guides/request-context) in static components.

## Dynamic rendering

A route declared `render: 'dynamic'` executes on every request. It sees current data, cookies, and headers.

```tsx
// ./src/pages/now.tsx
export default async function NowPage() {
  const now = new Date().toISOString();

  return <p>This page rendered at {now}.</p>;
}

export const getConfig = async () => {
  return {
    render: 'dynamic',
  } as const;
};
```

Waku does not place a cache in front of dynamic rendering or the data fetching inside it. What you fetch is what you render, so when your data changes, the next request sees it. There is no Waku revalidation step because there is nothing to invalidate.

Other systems on the request path may still cache on their own terms: database clients, external API SDKs, reverse proxies, CDNs, and browsers. Those are their caches with their own controls, not part of Waku's model.

## Static page with a dynamic slice

The modes compose within one page. A [slice](https://waku.gg/#slices) is a component with its own rendering mode, and a lazy slice is requested independently from the page it appears on. That combination puts a fresh region inside a prerendered shell:

```tsx
// ./src/pages/_slices/inventory.tsx
export default async function InventorySlice() {
  const stock = await getStock();

  return <p>{stock} in stock right now.</p>;
}

export const getConfig = async () => {
  return {
    render: 'dynamic',
  } as const;
};
```

```tsx
// ./src/pages/product.tsx
import { Slice } from 'waku';

export default async function ProductPage() {
  return (
    <div>
      <h1>Product</h1>
      <Slice id="inventory" lazy fallback={<p>Checking stock...</p>} />
    </div>
  );
}
```

The page is prerendered at build time with the fallback in place; the slice executes on each request and streams in. This is the shape for catalogs with live inventory, articles with personalized panels, and similar mostly-static pages.

## Explicit caching

Caching in Waku is an optimization you add deliberately, not a correctness requirement. Most applications should start without one: static handles the stable content, dynamic stays fresh on its own, and there is nothing to invalidate.

When measurement shows repeated, expensive dynamic work, you can cache exactly that work with the separate [`waku-cache`](https://github.com/wakujs/waku-cache) package (early and still evolving). Nothing is cached unless you wrap it: you choose the keys, the TTL, the storage, and when to invalidate. Because the cache is an explicit layer that you own, removing it changes performance, never correctness.

## Client navigation and prefetching

During client-side navigation, the router reuses prefetched RSC payloads of static routes so that moving between pages is fast. This is a browser-side navigation optimization: it does not create a server data cache, and dynamic routes still execute on the server per request. See [Navigation Prefetching](/guides/navigation-prefetching) and [Router Fetch Strategy](/guides/router-fetch-strategy) for the details and the experimental tuning APIs.

## HTTP and infrastructure caches

Several caches can exist around a Waku app. They are separate layers with separate controls:

- **Browser cache**: driven by HTTP headers on responses.
- **CDN or reverse proxy**: caches whatever your `Cache-Control` headers permit.
- **Static build output**: files produced by the build, typically served with long-lived caching by hosts.
- **Application cache**: the explicit layer described above, if you add one.
- **Client or SDK caches**: database drivers and API clients may cache internally.

Waku does not manage these for you, which also means none of them can silently change what your server code computes.

## Choosing

1. Can this UI stay the same until the next deployment? Use static rendering.
2. Does it depend on the request or need current data? Use dynamic rendering.
3. Is one region of a static page request-specific? Use a dynamic lazy slice.
4. Is repeated dynamic work measurably expensive? Add an explicit cache around only that work, with a store that matches your deployment topology.

## Common misconceptions

- **"Dynamic means uncached."** Dynamic means uncached _by Waku_. You can still cache expensive work explicitly, and infrastructure layers can cache responses if you tell them to.
- **"Static output is a cache."** Static output is a build artifact. It has no keys, no TTL, and no invalidation API; it is replaced wholesale by the next build.
- **"Client navigation reuse makes server data stale."** It only affects what the browser shows during navigation; the server's answer to a dynamic request is always computed fresh.
- **"Server components run once."** Static server components execute at build time; dynamic server components execute on every request. Neither runs in the browser.
- **"Every framework needs a revalidation model."** An invalidation responsibility appears when a cache is added. Waku without an explicit cache has nothing to invalidate.
