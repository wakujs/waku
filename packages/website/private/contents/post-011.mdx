We're happy to announce Waku v0.24 with a brand new API: the Slice Components API! This will replace the page part component api, along with unlocking brand new functionality... `lazy` Slice Components.

We heard the feedback around the Page Part components and feel like the Slices API will both solve a lot of the issues, while also unlocking brand new flexibility with render patterns!

## Components are re-useable across pages

One of the key issues with the Page Parts api is that each file is tied to a particular route. This is backwards from how we normally compose components. Usually the pattern is to write a component in some `_components` directory then import it to be used on each of your routes. Well... this is exactly how the slices api works!

```
 src/pages
 ├── _components
 │   ├── _slice-one.tsx
 │   └── _slice-two.tsx
 └── some-page.tsx
```

And some-page.tsx will be written something like this:

```tsx
import { Slice } from 'waku';

export default function SomePage() {
  return (
    <div>
      <Slice id="slice-one" />
      <Slice id="slice-two" />
    </div>
  );
}

export const getConfig = () => {
  return {
    render: 'static',
    slices: ['slice-one', 'slice-two'],
  };
};
```

Please notice: `slices` must be specified with each of the slice ids in order for waku to know how to bundle your page without any compiler magic.

The file convention for Slice components is `_slice${string}.tsx` (TypeScript string literal syntax). We will look for these anywhere in the `src/pages` directory. And by default, the slice id will be the filename without the `_` prefix.

An example for `_slice-one.tsx`:

```tsx
export default function SliceOne() {
  return <p>🍕</p>;
}

export const getConfig = () => {
  return {
    id: 'slice-one', // can be omitted if it can be taken from the file name
    render: 'static', // default is 'dynamic'
  };
};
```

We're really happy with the new approach and how it will allow simply importing and assembling your components like normal components.

## `lazy` Slice Components

`lazy` Slice Components are a new addition with the Slices API. They allow components to be requested separately from the page they are used on. This is useful for components that are expensive to render, or components that could be hosted separately (e.g. on a CDN).

If we re-use the example above, it could be written like this:

```tsx
import { Slice } from 'waku';

export default function SomePage() {
  return (
    <div>
      <Slice id="slice-one" />
      <Slice id="slice-two" lazy fallback={<p>Two is loading...</p>} />
    </div>
  );
}

export const getConfig = () => {
  return {
    render: 'static',
    slices: ['slice-one'], // Note: `slice-two` is lazy so it is not included with the page config
  };
};
```

This makes it so that now you can have a `dynamic` Slice component for `slice-two` here, while the rest of the page is static.
